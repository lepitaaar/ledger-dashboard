# 농협 경매 매입·공동재고·예상 이익 MVP

## 요약
- [농협 API 명세](./nonghyup_api_specification.md)의 경매낙찰내역을 매입 데이터로 수집한다.
- 실제 연동 구현은 `src/server/services/nonghyup-client.ts`와 `nonghyup-crypto.ts`를 기준으로 관리한다.
- 기존 `Transaction`은 거래처 판매 기록으로 유지한다.
- 출하자별 판매 추적 없이 농협 품목코드 `naLatc`를 상품 마스터에 연결해 공동재고로 관리한다.
- `trqt`는 연결된 상품 규격의 수량으로 1:1 사용하며 별도 환산은 하지 않는다.
- 이동평균 원가로 `예상 이익(매출총이익)`을 계산한다.
- 최초 수집일은 `2026-01-01`, 자동 동기화는 매일 오전 6시 KST로 고정한다.

## MVP 구현

### 농협 API 클라이언트
- Node 서버 전용 어댑터로 `POST https://newgp.nonghyup.com/naieSvc/xframe`을 호출한다.
- 제공 예정인 실제 동작 클라이언트 소스를 기준으로 PBKDF2, AES, RSA-OAEP, `NHDATA1`, 쿠키 처리 방식을 이식한다.
- 흐름은 CSRF·세션 발급 → RSA 공개키 조회 → 로그인 → 일자별 경매 조회 순서다.
- 로그인 응답의 `naBzplc`, `trmnAmnno`, `wmTrmnAmnno`를 조회 요청에 사용한다.
- JWT·CSRF·세션 오류 발생 시 세션 전체를 다시 만들고 해당 요청을 1회만 재시도한다.
- 계정 잠금·비밀번호·권한 오류는 자동 반복하지 않고 동기화를 실패 처리한다.
- 계정정보와 토큰은 환경변수와 메모리에서만 사용하며 DB·로그·브라우저로 전달하지 않는다.

### 동기화
- 농협 API가 일자 단위이므로 월 동기화 내부에서 날짜별 `ONBIEBY5005R0R01` 요청을 순차 실행한다.
- `pageNo=1`, `inqCn=20`부터 시작해 합계행을 제외한 수집 건수가 `cCnt`에 도달할 때까지 페이지를 조회한다.
- `wmcLatcnm === "합계"` 행은 저장하지 않고 상세 합계 검증에만 사용한다.
- 저장 고유키는 `경매일+사업장+거래구분+oslpNo+aucNo+naLatc`로 구성한다.
- 같은 날짜 재조회 시 upsert하고, 기존에는 있었지만 새 응답에서 사라진 건은 취소 상태로 전환한다.
- `cCnt`, 상세 수량 합계, 금액 합계가 맞지 않으면 해당 날짜 결과를 반영하지 않는다.
- 동기화는 한 번에 하나만 실행되도록 DB 잠금을 사용한다.
- 최초 실행은 `2026-01-01`부터 전일까지 월별로 순차 백필한다.
- 매일 오전 6시는 현재일 경매내역을 조회해서 디비에 추가한다.
- 매월 1일은 현재월과 전월의 경매내역을 조회해서 디비에 반영한다.
    - 만약 삽입된 데이터와 다르지않다면 업데이트 하지않는다.
    - 다르다면 기존데이터를 삭제하고 다시 삽입한다.
- 관리자 화면에서 특정 월과 오늘 날짜를 수동 재동기화할 수 있게 한다.
- 관리자 화면에서 특정 월과 일 부터 특정 날짜까지 수동 재동기화 할 수 있게 한다.

### 데이터 모델
- `AuctionPurchase`: 경매일, 사업장, 거래구분, 원표·경매번호, 농협 품목코드, 품목명, 출하자, 중량, 등급, 묶음, 크기, 수량, 단가, 금액, 비고, 상품 ID, 활성 상태를 저장한다.
- `AuctionSyncRun`: 대상 기간, 상태, 조회·신규·수정·취소·실패 건수, 오류와 실행시간을 저장한다.
- `AuctionProductMapping`: `사업장+거래구분+naLatc`와 기존 `Product`를 연결한다.
- `Transaction.productId`를 선택 필드로 추가해 판매 기록을 상품 마스터와 연결한다.
- 기존 판매 기록은 `상품명+규격`이 상품 마스터와 정확히 일치할 때만 자동 연결하고 나머지는 검토 대상으로 둔다.
- `InventoryMovement`에 매입·판매·반품의 수량 변화, 적용 원가, 재고수량, 이동평균 원가를 물질화한다.
- 같은 날짜에는 경매 매입을 먼저, 판매는 `registeredTimeKST` 순으로 계산한다.

### 재고와 이익
```text
새 이동평균 원가 =
(기존 재고금액 + 신규 경매 매입금액)
÷ (기존 재고수량 + 신규 경매 수량)

예상 이익 =
판매금액 - 판매수량 × 판매 시점 이동평균 원가
```
- 농협의 `selAm`을 매입금액의 기준으로 사용하고 `trqt × actoUpr` 불일치는 경고한다.
- 판매 반품은 음수 매출·음수 판매원가로 반영한다.
- 품목 매핑, 판매 상품 연결, 경매 수정·취소, 판매 수정·삭제 시 해당 상품 원장을 다시 계산한다.
- 재고가 부족해지는 상품은 그 시점부터 원가 계산을 중단하고 `기초재고 필요`로 표시한다.
- 기초재고는 `2026-01-01` 기준 수량과 평균원가를 선택적으로 등록한 뒤 재계산할 수 있다.
- 매핑되지 않았거나 원가 계산이 불가능한 판매는 이익 합계에서 제외하고 제외 건수·매출액을 함께 표시한다.

## 화면과 인터페이스
- **경매 매입:** 월별 내역, 매입 합계, 출하자·품목·매칭 상태 필터, 동기화 실행 및 이력.
- **품목 매칭:** 농협 품목코드를 기존 상품에 연결하거나 신규 상품을 생성.
- **재고 현황:** 상품·규격별 수량, 평균원가, 재고금액, 부족 여부.
- **손익 분석:** 기간별 매출, 판매원가, 예상 이익, 이익률, 원가 계산 포함률.
- 기존 거래 조회에도 상품 연결 상태와 예상 이익을 표시한다.
- API는 경매목록, 동기화, 품목매핑, 재고, 손익 조회로 분리하고 동기화 API는 `X-Sync-Secret`으로 보호한다.
- Docker 배포에는 오전 6시 KST에 내부 동기화 API를 호출하는 scheduler 서비스를 추가한다.

## 후속 기능
- 목표 이익률 기반 권장 판매가
- 품목별 매입·판매 단가 추이와 급등 경고
- 장기 재고 및 예상 소진일
- 폐기·증정·분실 재고 조정
- 출하자별 매입단가·구매량·품질 메모
- 거래처별·품목별 이익 기여도
- 월간 손익 리포트와 엑셀 출력

## 테스트 및 완료 기준
- 암호화·로그인·쿠키 처리는 제공되는 동작 소스의 고정 fixture로 단위 테스트한다.
- 토큰 만료, 네트워크 오류, 권한 오류에서 재시도와 실패 정책을 검증한다.
- 페이지네이션, 합계행 제외, 중복 방지, 수정·취소 동기화를 검증한다.
- 여러 출하자의 같은 상품이 하나의 공동재고로 합산되는지 확인한다.
- 이동평균, 판매, 반품, 재고 부족, 기초재고 재계산을 검증한다.
- 모든 날짜 경계와 오전 6시 실행은 `Asia/Seoul` 기준으로 검증한다.
- 라이브 농협 호출 테스트는 별도 환경변수가 있는 경우에만 수동 실행하며 CI에서는 제외한다.
- 대시보드에는 계산 가능한 이익과 제외된 매출이 구분되어야 한다.

## 문서와 보안 전제
- 현재 API 문서의 실제 JWT, 사용자 ID, 사업장·고객번호 등 식별값은 커밋 전에 예시값으로 치환한다.
- AES 키·IV 길이 설명, 로그인 평문 조합, `NHDATA1`, 쿠키 처리, `cCnt` 의미는 제공되는 동작 소스로 재검증해 문서를 보완한다.
- 농협 사이트 변경으로 암호화나 응답 계약이 달라지면 동기화 실패로 처리하고 기존 매입 데이터는 유지한다.
- 운송비·포장비·폐기비는 MVP에서 제외하므로 결과 명칭은 `순이익`이 아닌 `예상 이익(매출총이익)`으로 사용한다.
