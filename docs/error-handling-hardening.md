# 오류 처리 및 데이터 무결성 강화

기준: 2026-06-13

## 목표

- 부분 성공을 API 실패로 오인해 재시도하면서 데이터가 중복되는 문제를 방지한다.
- 재고 원장 재계산 중 장애가 발생해도 기존 원장을 보존한다.
- 외부 API, 스케줄러, 화면 조회 실패를 명시적으로 처리한다.
- 요청 단위 추적 정보와 일관된 오류 응답을 제공한다.

## MongoDB 운영 전제

본 데이터, 감사 로그, 재고 원장을 하나의 트랜잭션으로 처리한다. 운영 MongoDB는 트랜잭션을 지원하는 Atlas 클러스터 또는 replica set이어야 한다. standalone MongoDB에서는 쓰기 API가 실패한다.

배포 전 확인:

1. `MONGODB_URI`가 Atlas 또는 replica set을 가리키는지 확인한다.
2. 거래 생성 후 감사 로그와 재고 원장이 함께 생성되는지 확인한다.
3. 의도적으로 감사 로그 쓰기를 실패시켜 거래도 롤백되는지 스테이징에서 확인한다.

## 적용 내용

### 원자적 재고 원장 재계산

- 상품, 매입, 거래 원본을 트랜잭션 스냅샷에서 조회한다.
- 새 원장 계산을 메모리에서 완료한 뒤 기존 원장을 삭제한다.
- 삭제와 새 원장 저장은 같은 MongoDB 트랜잭션에서 실행한다.
- 원본 조회, 계산, 저장 중 하나라도 실패하면 기존 원장이 유지된다.

### 원자적 쓰기 API

다음 작업은 본 데이터, 감사 로그, 재고 재계산을 같은 트랜잭션으로 처리한다.

- 거래 생성, 수정, 삭제
- 계산서 관리 행 생성, 수정, 삭제
- 단건 반품과 벌크 삭제/반품
- 입금 등록
- 정산서 발행

벌크 작업은 전체 검증 후 한 트랜잭션에서 처리하므로 일부 행만 반영되지 않는다. 계산서 관리에서 거래를 삭제할 때도 영향받는 상품의 재고 원장을 재계산한다.

## 장애 대응

트랜잭션 미지원 오류가 발생하면 MongoDB 토폴로지를 먼저 확인한다. 데이터 쓰기 오류 후에는 클라이언트 재시도 전에 API 응답의 요청 ID로 서버 로그를 조회한다.

외부 연동과 화면 오류 처리 항목은 같은 문서에 후속 변경 내용을 계속 기록한다.
