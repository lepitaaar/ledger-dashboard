# 오류 처리 및 데이터 무결성 강화

기준: 2026-06-13

## 목표

- 부분 성공을 API 실패로 오인해 재시도하면서 데이터가 중복되는 문제를 방지한다.
- 재고 원장 재계산 중 장애가 발생해도 기존 원장을 보존한다.
- 외부 API, 스케줄러, 화면 조회 실패를 명시적으로 처리한다.
- 요청 단위 추적 정보와 일관된 오류 응답을 제공한다.

## MongoDB 운영 전제

본 데이터, 감사 로그, 재고 원장을 하나의 트랜잭션으로 처리한다. 운영 MongoDB는 트랜잭션을 지원하는 Atlas 클러스터 또는 replica set이어야 한다. standalone MongoDB에서는 쓰기 API가 실패한다.

배포 전 확인:

1. `MONGODB_URI`가 Atlas 또는 replica set을 가리키는지 확인한다.
2. 거래 생성 후 감사 로그와 재고 원장이 함께 생성되는지 확인한다.
3. 의도적으로 감사 로그 쓰기를 실패시켜 거래도 롤백되는지 스테이징에서 확인한다.

## 적용 내용

### 원자적 재고 원장 재계산

- 상품, 매입, 거래 원본을 트랜잭션 스냅샷에서 조회한다.
- 새 원장 계산을 메모리에서 완료한 뒤 기존 원장을 삭제한다.
- 삭제와 새 원장 저장은 같은 MongoDB 트랜잭션에서 실행한다.
- 원본 조회, 계산, 저장 중 하나라도 실패하면 기존 원장이 유지된다.

### 원자적 쓰기 API

다음 작업은 본 데이터, 감사 로그, 재고 재계산을 같은 트랜잭션으로 처리한다.

- 거래 생성, 수정, 삭제
- 계산서 관리 행 생성, 수정, 삭제
- 단건 반품과 벌크 삭제/반품
- 입금 등록
- 정산서 발행

벌크 작업은 전체 검증 후 한 트랜잭션에서 처리하므로 일부 행만 반영되지 않는다. 계산서 관리에서 거래를 삭제할 때도 영향받는 상품의 재고 원장을 재계산한다.

## 장애 대응

트랜잭션 미지원 오류가 발생하면 MongoDB 토폴로지를 먼저 확인한다. 데이터 쓰기 오류 후에는 클라이언트 재시도 전에 API 응답의 요청 ID로 서버 로그를 조회한다.

외부 연동과 화면 오류 처리 항목은 같은 문서에 후속 변경 내용을 계속 기록한다.

### MongoDB 연결 복구

- 최초 연결 Promise가 거부되면 연결과 Promise 캐시를 모두 비운다.
- MongoDB가 복구된 다음 요청은 새 연결을 시도한다.

### 농협 동기화 안정성

- 락에 실행별 `ownerId`를 기록해 다른 실행이 획득한 락을 이전 실행이 해제하지 못하게 한다.
- 5분 간격 heartbeat로 `lockedAt`을 갱신한다.
- 실행 로그 생성도 `try/finally` 범위에 포함해 로그 생성 실패 시 락을 해제한다.
- 농협 API 요청 기본 timeout은 15초이며 일시적 네트워크 오류, `408`, `429`, `5xx`는 최대 2회 시도한다.
- 스케줄러의 앱 요청 기본 timeout은 35분이다. 성공 응답을 받은 뒤에만 당일 완료로 표시하며, 실패하면 기본 5분 간격으로 재시도한다.

조정 가능한 환경 변수:

- `NONGHYUP_REQUEST_TIMEOUT_MS`
- `SYNC_REQUEST_TIMEOUT_MS`
- `SYNC_RETRY_INTERVAL_MS`

### 공통 API 오류 계약과 중복 요청 방지

오류 응답은 기존 `message`와 함께 다음 구조를 제공한다.

```json
{
  "message": "입력값 검증에 실패했습니다.",
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "입력값 검증에 실패했습니다.",
    "requestId": "uuid",
    "issues": {}
  }
}
```

- 응답 헤더 `X-Request-Id`와 본문의 `requestId`는 동일하다.
- Zod 검증, 잘못된 JSON, 잘못된 식별자, MongoDB duplicate key를 각각 `400`, `400`, `400`, `409`로 구분한다.
- 알 수 없는 서버 오류는 요청 ID를 포함한 JSON 구조 로그로 기록한다.
- 클라이언트 기본 timeout은 15초이며 `NEXT_PUBLIC_REQUEST_TIMEOUT_MS`로 조정한다.
- 거래, 입금, 정산서 발행, 계산서 행 생성, 단건 반품, 벌크 처리 POST는 `Idempotency-Key`를 지원한다.
- 지원되는 POST가 네트워크 단계에서 실패하면 클라이언트는 같은 키로 한 번 재시도한다. 완료된 키는 24시간 동안 같은 응답을 재생한다.
