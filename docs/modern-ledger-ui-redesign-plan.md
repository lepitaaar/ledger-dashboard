# 모던 장부 UI 개편 계획

## 1. 목표

현재 장부의 기능과 API는 유지하면서, 화면 구조와 공통 UI를 단계적으로 개편한다.

핵심 목표:

- 자주 쓰는 조회, 입력, 수정 작업의 클릭 수와 탐색 시간을 줄인다.
- 모든 화면에서 제목, 필터, 요약 지표, 표, 상태 표현 방식을 통일한다.
- 넓은 데이터 표의 가독성을 유지하면서 태블릿과 모바일에서도 핵심 업무를 처리할 수 있게 한다.
- 장식적인 대시보드보다 정확성, 상태 인지, 빠른 조작을 우선한다.

## 2. 현재 상태 진단

### 잘 되어 있는 부분

- Next.js, Tailwind CSS, Radix UI 기반의 공통 컴포넌트 구조가 이미 있다.
- Toast, Dialog, Combobox, DatePicker, Pagination, Skeleton 등 기본 업무 컴포넌트가 준비되어 있다.
- 거래, 정산, 거래처, 상품, 재고, 손익의 업무 영역이 URL과 화면 단위로 분리되어 있다.
- 금액 정렬, 엑셀 다운로드, 기간 필터 등 장부에 필요한 핵심 기능이 구현되어 있다.

### 개선이 필요한 부분

- 화면별 시각 언어가 다르다. 일부 화면은 단순 카드형이고, 일부는 강한 그라데이션 배너를 사용한다.
- 페이지 제목, 설명, 주요 액션의 위치가 화면마다 다르다.
- 사이드바가 `lg` 미만에서 완전히 사라져 모바일 탐색 수단이 없다.
- 거래 화면의 필터와 날짜 조건이 분리되어 현재 적용 조건을 한눈에 보기 어렵다.
- 여러 표가 직접 작성되어 열 높이, 헤더, 상태 배지, 빈 상태, 로딩 상태가 일관되지 않다.
- 기본 글자 크기가 `18px`이라 고밀도 장부 표에서 한 화면에 보이는 정보량이 적다.
- 영어 보조 문구와 화면별 강한 포인트 색상이 내부 장부 서비스의 일관성을 약화한다.
- 날짜와 기준일 일부가 화면 문구에 하드코딩되어 있다.
- `/dashboard`가 거래 조회로 바로 이동하여 전체 업무 상황을 요약하는 홈 화면이 없다.

## 3. 디자인 방향

### 콘셉트

`신뢰할 수 있는 업무용 금융 대시보드`

- 배경: 밝은 회색 계열
- 표면: 흰색 카드와 얇은 경계선
- 주 색상: 네이비 또는 짙은 블루 한 가지
- 상태 색상: 성공은 초록, 주의는 주황, 오류와 음수는 빨강
- 과도한 그라데이션과 그림자는 줄이고 숫자, 상태, 작업 버튼의 위계를 강화한다.

### 타이포그래피

- 본문 기본 크기: `16px`
- 데이터 표: `14px`
- 보조 정보: `12px`
- 페이지 제목: `24px`에서 `28px`
- 금액과 수량은 우측 정렬하고 `tabular-nums`를 적용한다.
- 영문 장식 라벨은 제거하고 업무 용어를 한국어로 통일한다.

### 밀도

- 기본은 편안한 밀도, 표에는 `compact` 밀도를 적용한다.
- 입력 컨트롤 높이는 기본 `40px`, 작은 필터 버튼은 `32px` 또는 `36px`로 통일한다.
- 카드 안쪽 여백은 화면별 임의 지정 대신 공통 규칙을 사용한다.

- 상단 바에는 현재 페이지명, 오늘 날짜, 전역 빠른 작업 영역을 배치한다.

## 5. 공통 화면 구조

모든 목록 및 분석 화면을 다음 순서로 통일한다.

1. `PageHeader`: 제목, 한 줄 설명, 주요 액션
2. `FilterBar`: 검색, 기간, 선택 필터, 초기화
3. `MetricGrid`: 화면에 꼭 필요한 KPI만 표시
4. `DataCard`: 표, 차트 또는 상세 콘텐츠
5. `Pagination`: 건수, 현재 범위, 페이지 이동

필요한 공통 컴포넌트:

- `AppShell`
- `DesktopSidebar`
- `MobileNavigation`
- `PageHeader`
- `FilterBar`
- `ActiveFilterChips`
- `MetricCard`
- `DataTableShell`
- `StatusBadge`
- `EmptyState`
- `ErrorState`
- `TableToolbar`
- `MoneyText`

## 6. 화면별 개편안

### 대시보드 홈

신규 `/dashboard` 화면을 만들고 리다이렉트를 제거한다.

- 오늘 매출
- 이번 달 매출
- 미수금
- 재고 부족 품목
- 최근 거래
- 처리가 필요한 품목 매칭
- `거래 등록`, `계산서 관리`, `입금 기록` 빠른 작업

초기 버전은 기존 API를 조합해 구현하고, 성능 문제가 확인되면 대시보드 전용 집계 API를 추가한다.

### 거래 조회

- 상단에 기간 매출, 거래 건수, 평균 거래액을 표시한다.
- 기간 프리셋, 날짜 범위, 거래처, 상품, 검색어를 한 필터 바에 배치한다.
- 적용된 조건은 칩으로 표시하고 한 번에 초기화할 수 있게 한다.
- 표 헤더를 한 줄 중심으로 단순화하고 헤더를 스크롤 시 고정한다.
- 금액, 수량, 이익률은 우측 정렬한다.
- 상품 매칭 상태는 점 문자가 아닌 공통 상태 배지를 사용한다.
- 모바일에서는 중요 열만 카드형 행으로 보여주고 상세 정보는 행 확장으로 제공한다.

### 계산서 및 상세 관리

- 계산서 목록과 상세 편집의 관계를 명확히 한다.
- 저장 상태, 미저장 변경, 오류 상태를 상단 고정 영역에 표시한다.
- 행 추가, 반품, 삭제, 인쇄, 엑셀 액션의 중요도와 색상을 구분한다.
- 편집 표는 첫 열과 작업 열을 고정하고 키보드 이동 흐름을 점검한다.
- 파괴적 작업에는 대상과 영향 범위를 확인하는 문구를 사용한다.

### 거래처 관리

- 전체 거래처 수만 표시하는 큰 KPI 카드는 제거하거나 활성 거래처, 미수 거래처와 함께 묶는다.
- 검색과 신규 등록을 페이지 헤더에 배치한다.
- 업체명, 연락처, 이번 달 거래액, 미수금, 상태, 작업을 핵심 열로 구성한다.
- 거래처 상세에서는 매출, 입금, 미수금의 관계를 가장 먼저 보여준다.
- 입금 기록 추가 폼은 별도 카드보다 명확한 주요 액션과 다이얼로그 사용을 검토한다.

### 상품 및 재고

- 상품 관리와 재고 현황의 용어 및 상태 표시를 통일한다.
- 재고 부족, 미매칭, 기초재고 필요를 같은 상태 체계로 표현한다.
- 기초재고 기준일은 하드코딩하지 않고 서버 설정 또는 데이터 기준일을 표시한다.
- 재고 표에 부족 품목 필터와 정렬을 추가한다.

### 경매 매입 및 품목 매칭

- 강한 그라데이션 헤더를 공통 페이지 헤더로 교체한다.
- 동기화 상태와 마지막 동기화 시각을 상단 상태 영역에 배치한다.
- 미매칭 항목을 우선 처리할 수 있도록 기본 정렬과 필터를 제공한다.
- 동기화 실행은 중복 실행 방지와 진행 상태를 명확히 보여준다.

### 손익 분석

- 장식 요소를 줄이고 기간, 매출, 원가, 이익, 마진의 관계를 우선한다.
- 단순 SVG 게이지보다 비교가 쉬운 막대 또는 추세 차트를 사용한다.
- 원가 반영률이 낮을 때 결과의 신뢰도 경고를 KPI 가까이에 표시한다.
- 수익과 손실은 색상뿐 아니라 부호와 라벨로도 구분한다.

## 7. 구현 단계

### Phase 0. 기준 확정

- 대표 화면 3개를 선정한다: 거래 조회, 거래처 관리, 재고 현황.
- 데스크톱 `1440px`, 태블릿 `768px`, 모바일 `390px`을 검수 기준으로 정한다.
- 색상, 글자 크기, 간격, 모서리, 그림자 토큰을 확정한다.

완료 조건:

- 한 페이지 분량의 UI 원칙과 대표 화면 와이어프레임이 합의되어 있다.

### Phase 1. 디자인 시스템 정리

- `globals.css`와 Tailwind 토큰을 정리한다.
- 전역 기본 글자 크기를 `16px`로 조정한다.
- Button, Card, Table, Badge, Input의 크기와 상태를 통일한다.
- `PageHeader`, `MetricCard`, `FilterBar`, `StatusBadge`, `EmptyState`를 만든다.
- Storybook을 새로 도입하지 않고 실제 대표 화면에서 컴포넌트를 검증한다.

완료 조건:

- 공통 컴포넌트가 키보드 포커스, 비활성, 로딩, 오류 상태를 지원한다.

### Phase 2. 앱 셸과 반응형 탐색

- `dashboard/layout.tsx`를 `AppShell` 구조로 변경한다.
- 사이드바에 메뉴 그룹과 아이콘을 추가한다.
- 모바일 드로어 메뉴와 상단 바를 추가한다.
- `/dashboard` 홈 화면을 구현한다.

완료 조건:

- 모든 화면에 데스크톱과 모바일 탐색 수단이 있다.
- 현재 메뉴와 페이지 제목을 즉시 파악할 수 있다.

### Phase 3. 핵심 장부 흐름 개편

- 거래 조회
- 계산서 목록
- 계산서 상세 관리
- 거래처 목록 및 상세

이 단계에서 필터 바, 상태 배지, 표 밀도, 빈 상태, 로딩 상태를 공통화한다.

완료 조건:

- 조회 조건과 적용 결과가 명확하다.
- 주요 작업을 데스크톱과 모바일에서 수행할 수 있다.
- 기존 조회, 저장, 반품, 삭제, 다운로드, 인쇄 기능이 유지된다.

### Phase 4. 상품, 재고, 매입 개편

- 상품 관리
- 재고 현황
- 경매 매입
- 품목 매칭

완료 조건:

- 재고 및 매칭 문제 항목을 필터링하고 우선 처리할 수 있다.
- 동기화와 기준일 상태가 화면에 명확히 표시된다.

### Phase 5. 분석 화면 개편

- 손익 분석의 KPI와 차트를 재구성한다.
- 기간 비교 또는 추세 데이터가 필요하면 API 범위를 별도로 설계한다.
- 분석 결과의 원가 커버리지와 주의 문구를 강화한다.

완료 조건:

- 사용자가 매출, 원가, 이익, 마진, 데이터 신뢰도를 한 화면에서 이해할 수 있다.

### Phase 6. 품질 검증

- `npm run typecheck`
- `npm run lint`
- `npm run test`
- 주요 업무 플로우 수동 회귀 테스트
- 키보드 탐색 및 포커스 표시 점검
- 색상 대비 점검
- `390px`, `768px`, `1440px` 화면 검수
- 긴 업체명, 큰 금액, 데이터 없음, API 오류, 로딩 상태 검수

## 8. 권장 작업 순서

첫 구현 단위는 다음 세 묶음으로 진행한다.

1. 앱 셸, 디자인 토큰, 공통 페이지 헤더
2. 거래 조회 화면을 기준 화면으로 개편
3. 거래처 관리와 재고 현황에 같은 패턴 확장

이 세 단계가 끝나면 디자인 방향과 공통 컴포넌트의 실효성을 확인한 뒤 나머지 화면으로 확장한다.

## 9. 범위에서 제외할 항목

초기 UI 개편에서는 다음 항목을 분리한다.

- 데이터 모델 전면 변경
- 정산 및 재고 계산 로직 변경
- 인증과 권한 체계 신규 도입
- 모바일 네이티브 앱 개발
- 새로운 차트 라이브러리의 선제적 도입

UI 개편 중 필요성이 확인된 API 변경은 화면 단위로 최소 범위만 추가한다.
