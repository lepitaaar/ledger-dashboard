# 계산서 입력 및 거래처 UX 개선 계획

기준: 2026-06-06

## 1. 목적

데스크톱 장부 관리 업무에서 계산서 품목을 반복 입력할 때 저장 상태를 명확히 전달하고, 다중 행 삭제·반품 작업의 효율성과 안전성을 높인다.

거래처 대표자명과 전화번호는 선택 입력 데이터로 유지하되, 일상적인 운영 화면에서는 노출하지 않아 필요한 정보에 집중할 수 있도록 한다.

## 2. 기본 원칙

- 모바일 UI는 지원 범위에서 제외한다.
- 계산서 관리 화면의 기본 15개 입력 행을 유지한다.
- 저장 방식은 행 단위 자동 저장을 사용한다.
- 거래처 데이터 모델과 기존 대표자명·전화번호 데이터는 삭제하지 않는다.
- 기존 엑셀 저장과 A5 거래명세서 인쇄 동작을 유지한다.

## 3. 계산서 입력 UX

적용 화면:
- `/dashboard/settlements/manage`

행 상태:
- 저장된 행은 기본적으로 읽기 상태로 표시한다.
- 저장된 행은 `수정` 버튼을 눌러 해당 행만 편집 상태로 전환한다.
- 신규 빈 행은 별도 조작 없이 바로 입력할 수 있다.
- 행마다 `변경됨`, `저장 중`, `저장 완료`, `저장 실패` 상태를 표시한다.

키보드 입력:
- `Tab`으로 같은 행의 다음 필드로 이동한다.
- 마지막 입력 필드에서 `Enter`를 누르면 해당 행을 저장한다.
- 저장된 행을 수정하다가 `Escape`를 누르면 마지막 저장값으로 복원한다.

자동 저장:
- 품목, 수량, 단가 등 필수값이 완성된 상태에서 행 밖으로 이동하면 저장한다.
- 미완성 행은 서버에 저장하지 않고 잘못된 필드에 오류를 표시한다.
- 저장 성공 후 전체 목록을 다시 조회하지 않고 API 응답으로 해당 행만 갱신한다.
- 저장 실패 시 입력값과 포커스를 유지하고 재시도 기능을 제공한다.
- 저장 중에도 다른 행의 입력 흐름이 불필요하게 초기화되지 않아야 한다.

업체·날짜 이동:
- 미저장 변경이나 저장 실패 행이 있으면 업체 또는 날짜를 변경하기 전에 경고한다.
- 사용자가 이동을 취소하면 현재 입력값과 선택 상태를 유지한다.

합계:
- 저장된 거래의 합계와 현재 편집 중인 예상 합계가 다르면 이를 구분해 표시한다.
- 삭제·반품·저장 성공 후 합계를 즉시 갱신한다.

## 4. 선택 및 일괄 작업

- 전체 선택과 개별 체크박스를 실제 일괄 작업에 연결한다.
- 서버에 저장된 행만 선택할 수 있다.
- 선택된 행 수를 작업 표시줄에 표시한다.
- 작업 표시줄에 `선택 삭제`와 `선택 반품`을 제공한다.
- 삭제와 반품을 실행하기 전에 대상 건수와 합계금액을 확인 Dialog에 표시한다.
- 음수 수량인 반품 거래는 다시 반품할 수 없다.
- 일괄 처리 중에는 중복 실행을 차단한다.
- 처리 결과는 Toast로 알리고 성공한 행과 합계를 화면에 반영한다.

## 5. 거래처 정보 노출

대표자명과 전화번호는 등록·수정 시 선택적으로 관리할 수 있지만 다음 운영 화면에서는 표시하지 않는다.

거래처 목록:
- 대표자명 열 제거
- 전화번호 열 제거
- 검색 안내 문구를 `업체명 검색`으로 변경

거래처 상세:
- 헤더에서 대표자명과 전화번호 제거
- 업체명과 거래 상태만 표시

계산서 업체 선택:
- 업체 카드에서 대표자명과 전화번호 제거
- 업체명만 표시

유지 사항:
- 거래처 등록·수정 Dialog의 대표자명과 전화번호 입력은 선택 항목으로 유지
- 기존 거래처 연락처 데이터와 API 필드는 호환성을 위해 유지
- 기존 API의 대표자명·전화번호 검색 동작은 유지
- 공급자 대표자명과 전화번호는 거래처 정보와 별개이므로 A5 거래명세서에 계속 표시

## 6. API 변경

### 일괄 처리 API

`POST /api/settlements/manage/bulk`

요청:

```json
{
  "action": "delete",
  "transactionIds": ["<transactionId>"]
}
```

규칙:
- `action`은 `delete` 또는 `return`
- 중복 ID 제거
- 요청당 최대 100개
- 처리 전 모든 대상이 존재하고 삭제되지 않았는지 검증
- `return`은 수량이 양수인 거래만 허용
- 대상 검증이 실패하면 실제 변경을 시작하지 않음

응답:

```json
{
  "data": {
    "action": "delete",
    "processedIds": ["<transactionId>"],
    "createdIds": []
  }
}
```

- 반품 처리 시 생성된 음수 거래 ID를 `createdIds`로 반환한다.
- 삭제·반품 결과는 기존 감사 로그에 행별로 기록한다.

### 기존 API

- 단건 생성·수정 응답으로 클라이언트의 해당 행을 직접 갱신한다.
- 기존 단건 삭제·반품 API는 호환성을 위해 유지한다.
- 거래처 연락처 관련 API와 데이터 필드는 제거하지 않는다.

## 7. 공통 피드백

- 브라우저 기본 `alert`와 `confirm`을 Toast 및 Alert Dialog로 교체한다.
- 수정, 반품, 삭제 등 아이콘 버튼에 Tooltip과 접근 가능한 이름을 제공한다.
- 계산서 화면의 로딩, 빈 상태, 오류 상태 표현을 일관되게 정리한다.
- 오류 메시지는 작업 대상과 재시도 방법을 사용자가 파악할 수 있게 표시한다.

## 8. 테스트 계획

계산서 입력:
- 신규 행의 행 이탈 자동 저장
- 마지막 필드 `Enter` 저장
- `Tab` 입력 순서
- 저장된 행 수정 및 `Escape` 취소
- 저장 실패 후 값·포커스 유지와 재시도
- 불완전한 행의 미저장 및 필드 오류 표시
- 저장 중 다른 행 입력 상태 유지

일괄 작업:
- 전체 선택, 일부 선택, 선택 해제
- 저장되지 않은 빈 행 선택 제한
- 선택 행 일괄 삭제와 합계 갱신
- 양수 거래 일괄 반품
- 음수 거래의 반품 선택 제한
- 존재하지 않거나 삭제된 ID가 포함된 요청 거부
- 처리 중 중복 실행 차단

거래처:
- 목록에서 대표자명·전화번호 미노출
- 상세 헤더에서 대표자명·전화번호 미노출
- 계산서 업체 카드에서 대표자명·전화번호 미노출
- 등록·수정 Dialog에서 선택 입력 유지

회귀 검증:
- 엑셀 저장
- A5 거래명세서 단일·다중 페이지 인쇄
- `npm run typecheck`
- `npm run lint`
- `npm run test`
- `npm run build`
- 1440px 데스크톱 브라우저에서 주요 흐름 확인
